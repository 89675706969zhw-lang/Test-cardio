# Обновлённое описание ИС кардиореабилитации

## Выявленные недостатки в исходном описании

1. Смешивались исследовательский и клинический контуры без чёткой границы между обучением и применением модели у пациента.
2. Не была формально задана процедура выбора числа классов пациентов (кластеров), из-за чего сегментация оставалась субъективной.
3. Не определялись метрики качества для моделей по каждому классу и критерии допуска модели к использованию.
4. Цифровой двойник описывался концептуально, но без операционного цикла: инициализация состояния, шаг симуляции, функция цели, ограничения безопасности.
5. Контур обновления моделей не имел воспроизводимого протокола (версии данных/моделей, сравнение до/после).
6. Формирование персонализированного плана не было привязано к шаблону документа и не было сведено к программному синтезу.

## Исправленное архитектурное решение

### 1) Контуры системы
- **Клинический контур (on-prem):** сбор данных пациента, запуск инференса модели соответствующего класса, выдача рекомендаций врачу.
- **Исследовательский контур:** кластеризация, обучение/переобучение, виртуальные эксперименты на цифровых двойниках, сравнение версий моделей.

### 2) Данные и признаки
- Используются 40 признаков датасета, включая кардиогемодинамику, функциональные тесты, лабораторию, frailty/нутрицию, психоэмоциональные шкалы.
- Целевая переменная: `Успех_реабилитации_01`.

### 3) Оптимальное разделение пациентов на классы
- Применяется `KMeans` на стандартизированных признаках.
- Число классов выбирается по максимуму `silhouette score` в диапазоне K=2..6.
- В выполненном эксперименте оптимум: **K=2**.

### 4) Модели по классам
- Для каждого класса обучается отдельная модель классификации успеха реабилитации.
- Кандидаты: `LogisticRegression`, `RandomForest`, `GradientBoosting`.
- Выбор лучшей модели класса выполняется по `ROC-AUC` на отложенной выборке класса.

### 5) Цифровой двойник класса
- Для каждого класса выбирается репрезентативный пациент (ближайший к центроиду).
- Двойник работает как симулятор на горизонте 12 недель.
- На каждом шаге моделируются изменения ключевых показателей (`6MWT`, `TUG`, `QoR15`, SpO2 и др.) в ответ на параметры плана (интенсивность, частота, длительность).
- Введены safety-ограничения (SpO2, лактат, ЧСС) и штрафы за нарушения.

### 6) Виртуальные эксперименты и обновление модели
- Для каждого класса выполняется перебор сетки допустимых планов.
- Оптимизация ведётся по многокритериальной функции: вероятность успеха + функциональный выигрыш - штрафы безопасности.
- После выбора лучшего плана генерируются синтетические траектории и выполняется обновление модели класса.
- Фиксируются метрики до/после (`AUC`, `F1`) для контроля эффекта обновления.

### 7) Программный синтез персонализированных планов
- Для каждого класса автоматически формируется план в структуре шаблона:
  - общая информация,
  - 4 фазы реабилитации,
  - правила персонализации,
  - алгоритм генерации,
  - целевые KPI на 12 недель.
- Количество планов равно числу классов.

## Фактически полученные результаты (на предоставленном датасете)

- Размер данных: 8992 пациентов, 40 признаков.
- Оптимальное число классов: **2**.
- Распределение по классам: класс 0 — 3697, класс 1 — 5295.
- Качество моделей:
  - класс 0: ROC-AUC 0.671, F1 0.598;
  - класс 1: ROC-AUC 0.674, F1 0.613.
